# Concepts

This page defines the main concepts used by the root Agentic Retrieval pipeline. It covers the general JSON and JSONL workflow configured from `config.yaml.example`.

## Pipeline Overview

Agentic Retrieval has two main stages:

1. **Ingestion** reads source documents, builds embedding vectors, and writes documents plus vectors to Azure Cosmos DB.
2. **Retrieval and answering** reads a question, retrieves relevant documents with vector and optional full-text search, optionally reduces or reranks those results, and asks an LLM to produce an answer.

The main scripts are `cosmos_db_upload.py` for ingestion and `dynamic_retriever.py` for retrieval and answering.

## Documents, Chunks, and Context

A **document** is one JSON object loaded from a configured source. The default loader in `cosmos_db_upload.py` supports JSONL files, where each line is a document. Other formats (for example, XML used by PubMed) are handled by setting a per-source `document_parser` and `file_glob` so the upload script knows how to discover and parse files in `documents_root`.

A **chunk** is the unit passed into the answer-generation prompt after retrieval. In this repo, a retrieved chunk is usually a formatted version of one Cosmos DB item rather than a separately split passage. The retriever removes system fields and embedding vectors, formats useful fields as text, and preserves configured metadata.

`context_fields` are source-level fields that should always appear in retrieved context if they exist in the document. Use them for identifiers or labels that help interpret a result, such as `doc_id`, `product_id`, or another stable reference.

## Embeddings

An **embedding** is a numeric vector representation of data, typically generated by a specialized model. During ingestion, the upload script concatenates the configured `embedding_text_fields`, sends that text to the configured embedding model, and stores the returned vector in the document field named by `embedding_field`.

For the sample config, `embedding_field` is `e`, so vectors are stored at JSON path `/e`. If you change the field, keep these settings aligned:

- `cosmos.sources[].embedding_field`
- `cosmos.vector_embedding_policy_json.vectorEmbeddings[].path`
- `cosmos.sources[].indexing_policy_json.vectorIndexes[].path`
- the matching excluded path, such as `/e/*`

## Azure Cosmos DB

[Azure Cosmos DB](https://learn.microsoft.com/en-us/azure/cosmos-db/) is the database used by this repo to store source documents and their embedding vectors. The root pipeline uses the Azure Cosmos DB for NoSQL API with document-shaped items, SQL-like queries, vector search, and optional full-text search.

A **Cosmos DB account** is the Azure resource endpoint configured by `cosmos.uri`. A **database** is the logical database named by `cosmos.database_name`. A **container** is the collection that stores items for one source and owns settings such as partition key, indexing policy, vector policy, full-text policy, and throughput.

## Sources

`cosmos.sources` is the central mapping between local data and Cosmos DB containers. Each source describes one container and its retrieval behavior.

Important source fields include:

- `id`: internal source name used in retrieval metadata.
- `container_name`: Cosmos DB container to create or use.
- `partition_key_path`: JSON path for the container partition key.
- `embedding_field`: document field that stores embedding vectors.
- `documents_root`: local file or folder used during upload.
- `embedding_text_fields`: document fields concatenated for embedding input.
- `retrieval.vector_k`: number of vector results to retrieve from this source.
- `retrieval.fulltext_k`: number of full-text results to retrieve from this source.
- `retrieval.fulltext_fields`: fields searched by full-text retrieval.
- `indexing_policy_json`: Cosmos DB indexing policy for the container.
- `full_text_policy_json`: Cosmos DB full-text policy for the container.

Multi-source retrieval means the same question can query several containers, each with its own vector and full-text limits.

## Partition Keys

A **partition key** controls how items are distributed inside an Azure Cosmos DB container. Microsoft Learn describes the partitioning model in [Partitioning and horizontal scaling in Azure Cosmos DB](https://learn.microsoft.com/en-us/azure/cosmos-db/partitioning).

In this repo, each source sets `partition_key_path`. The sample config uses `/pk`. A good partition key should be present on every item and should distribute data well. For small sample datasets this is mostly setup plumbing; for larger datasets it affects throughput, storage distribution, and query cost.

## Indexing Policies

An **indexing policy** tells Azure Cosmos DB which fields and specialized indexes to maintain for a container. Microsoft Learn covers these settings in [Indexing policies in Azure Cosmos DB](https://learn.microsoft.com/en-us/azure/cosmos-db/index-policy).

The sample config includes:

- `includedPaths` to index normal fields.
- `excludedPaths` to skip system metadata.
- `fullTextIndexes` for fields that can be searched lexically.
- `vectorIndexes` for fields that can be queried with vector similarity.

Vector and full-text indexes must match the paths configured in the vector embedding policy and full-text policy.

## Vector Search

[Vector search in Azure Cosmos DB for NoSQL](https://learn.microsoft.com/en-us/azure/cosmos-db/nosql/vector-search) finds documents whose embedding vectors are close to the query embedding. In this repo, `dynamic_retriever.py` embeds the question, then `utils/cosmos_retriever.py` queries Cosmos DB with `VectorDistance` against each source's `embedding_field`.

`retrieval.vector_k` controls how many vector results are requested from a source before cross-source merging, diversity selection, and optional reranking.

The sample config uses a `diskANN` vector index. DiskANN is an approximate nearest-neighbor index type intended for efficient vector search at scale.

## Full-Text Search

[Full-text search in Azure Cosmos DB for NoSQL](https://learn.microsoft.com/en-us/azure/cosmos-db/gen-ai/full-text-search) searches text fields with lexical matching and scoring. It is useful when exact words, titles, names, or domain terms matter.

In this repo, `retrieval.fulltext_k` controls how many full-text results are requested per source. `retrieval.fulltext_fields` lists the document fields searched by the full-text retriever.

Set `fulltext_k: 0` or leave `fulltext_fields` empty to disable full-text retrieval for a source.

## Multi-Source Retrieval

A **source** can have vector retrieval, full-text retrieval, or both. The combined retriever runs enabled strategies for all configured sources, deduplicates results by source and document id, formats documents into chunks, and returns a shared list for downstream selection.

This is useful when a question may need evidence from different containers or when different document types need different retrieval settings.

## Diversity Selection

**Diversity selection** reduces a larger candidate set to a smaller set that covers more distinct information. This repo uses a greedy log-det selection algorithm in `greedy_log_det.py`.

`retrieval.k_diverse` controls the number of chunks selected after initial retrieval. Set it to `0` to disable diversity selection.

`retrieval.eta` and `retrieval.rescale_power` tune how the selection balances similarity to the query against diversity among retrieved embeddings.

## Semantic Reranking

The optional [Semantic Reranker in Azure Cosmos DB for NoSQL](https://learn.microsoft.com/azure/cosmos-db/gen-ai/semantic-reranker) (currently in Preview) can reorder retrieved chunks after diversity selection by scoring each candidate against the question with an AI model. It is configured under `ranker` in the YAML file.

Minimum settings to enable reranking:

- `ranker.use_ranker: true` to turn the stage on.
- `ranker.account_name` and `ranker.region` to address the ranker service.
- A token or identity-based access path so the client can authenticate.
- `ranker.k_ranker` to set how many chunks remain after reranking; this runs *after* diversity selection, so `k_ranker` should be `<= k_diverse` when both are enabled.

Leave `use_ranker: false` if any of these are missing.

## Decomposed RAG

**RAG** means retrieval-augmented generation: retrieve relevant context first, then ask an LLM to answer using that context.

This repo uses a decomposed RAG loop in `DecomposedRAGPipeline`. For each question, the pipeline:

1. Retrieves an initial context set and asks the LLM for a **preliminary answer**.
2. Performs **gap analysis** on that answer to identify missing or weakly supported claims.
3. Generates **sub-questions** targeting those gaps and retrieves additional context for them.
4. Repeats steps 2–3 until no useful gaps remain or the round limit is hit.
5. Synthesizes a final answer from the accumulated context and sub-answers.

Key settings that shape this loop:

- `pipeline.rounds`: maximum number of gap-analysis / sub-question loops per question. One round can produce multiple sub-questions.
- `pipeline.max_sub_questions`: maximum sub-questions generated in a single round.
- `pipeline.subq_fanout_cap`: hard cap on the total number of sub-question paths pursued across the whole run for one question (limits fan-out across rounds).
- `pipeline.subq_max_concurrency`: how many sub-question retrieval/LLM calls run in parallel.
- `pipeline.dataset_description`: optional description included in prompts to help the LLM ground its decomposition.

## Prompts

Prompt templates live in `prompts.py`. The pipeline uses them to ask for preliminary analysis, sub-question generation, gap analysis, and final answer synthesis.

The config fields `pipeline.preliminary_prefix` and `pipeline.subquery_prefix` let you add short dataset-specific instructions without editing the prompt source.

## Runtime Overrides

Many config values can be set directly on `dynamic_retriever.py` for a single run:

- `--k-diverse` # Number of retrieved chunks kept after diversity selection. Use `0` to skip diversity selection. Config: `retrieval.k_diverse`.
- `--k-ranker` # Number of chunks kept after semantic reranking. Use `0` to skip ranker reduction. Config: `ranker.k_ranker`.
- `--eta` # Regularization value used by diversity selection. Config: `retrieval.eta`.
- `--rescale-power` # How strongly query similarity influences diversity selection. Config: `retrieval.rescale_power`.
- `--max-sub-questions` # Maximum sub-questions the pipeline can generate in one round. Config: `pipeline.max_sub_questions`.
- `--subq-fanout-cap` # Limit on how many follow-up sub-question paths can be pursued. Config: `pipeline.subq_fanout_cap`.
- `--subq-max-concurrency` # Number of sub-question retrieval/generation tasks allowed to run at the same time. Config: `pipeline.subq_max_concurrency`.
- `--rounds` # Maximum number of retrieval and refinement rounds for a question. Config: `pipeline.rounds`.
- `--max-questions` # Limit on how many questions are processed from the questions file. Config: `execution.max_questions`.
- `--max-workers` # Number of questions that can be processed concurrently. Config: `execution.max_workers`.
- `--questions-path` # Question file selected for the run. Config: `paths.questions_path`.
- `--output-root` # Directory where run outputs and traces are written. Config: `paths.output_root`.
- `--timing` # Enables timing checkpoints and query logging for profiling a run.

Runtime overrides are useful for smoke tests and tuning experiments without editing YAML.

## Local LLM Fallback

The pipeline can offload the many small LLM calls used during decomposition (sub-question generation, gap analysis, and sub-answer drafting) to a local model, keeping the hosted Azure OpenAI model for the main answer. This is controlled by the `local_llm` section:

- `local_llm.local_fallback_endpoint`: HTTP endpoint of the local model, for example an Ollama generate URL.
- `local_llm.local_fallback_model`: model name to request from that endpoint. Leave empty to disable the fallback even if other fields are set.
- `local_llm.use_local_fallback_for_subtasks`: when `true`, sub-task LLM calls go to the local model; the final synthesis call still uses the configured Azure OpenAI deployment.
- `local_llm.local_max_concurrency`: maximum number of concurrent requests sent to the local endpoint.

This is useful for reducing cost and rate-limit pressure on the hosted model during runs with many sub-questions.

## Authentication and RBAC

The repo can use key-based authentication or Microsoft Entra ID authentication. For Cosmos DB data-plane access, see Microsoft Learn's guide to [connect to Azure Cosmos DB for NoSQL using role-based access control and Microsoft Entra ID](https://learn.microsoft.com/en-us/azure/cosmos-db/how-to-connect-role-based-access-control?pivots=azure-cli).

For Cosmos DB, `cosmos.use_rbac_auth: true` uses identity-based auth. For Azure OpenAI and embeddings, `llm.use_rbac_auth` and `embedding.use_rbac_auth` control whether the clients use identity-based auth or API keys.

## Outputs and Traces

Retrieval outputs are written under `paths.output_root` (overridable with `--output-root`). Each run lands in a subfolder whose name encodes the retrieval settings, so runs with different parameters do not collide:

```text
<output-root>/k{total_k}_ft{total_fulltext_k}_vec{total_vector_k}[_div{k_diverse}][_rank{k_ranker}]/
```

Within that folder:

- `intermediate/<group>/<question_id>.json` — per-question trace including retrieved chunks, sub-questions, and intermediate answers. `<group>` matches the questions file the question came from.
- `questions_with_answers.json` — one final answers file per input questions file, written at the end of the run.
- Timing logs, when `--timing` is set or when `timing_summary.py` is run against the output.

These files are useful for debugging which evidence was retrieved, which sub-questions were asked, and where time was spent.
